# SPEC: core/color_engine.py

> **Слой**: Core  
> **Зависимости**: numpy, OpenCV (cv2) — только для математических операций, ZERO I/O  
> **Ответственность**: Пиксельная замена цветов в изображении с поддержкой tolerance и антиалиасинг-блендинга  
> **Соседний файл**: `color_engine.py`

---

## 1. НАЗНАЧЕНИЕ

Ядро всей системы. Принимает numpy-массив RGBA и список маппингов (from_hex → to_hex) с tolerance. Возвращает новый numpy-массив с заменёнными цветами. Используется как для preview-replace (на уменьшенном изображении), так и для полной batch-обработки (на оригинале).

**Критически важно**: Качество замены определяет, оставит ли клиент отзыв. Ореолы, артефакты, потеря прозрачности — это провал.

---

## 2. КОНСТАНТЫ

- `MAX_DELTA_E`: float = 50.0 — максимальное расстояние Delta-E для шкалы tolerance
- Используется для пересчёта tolerance (0–100) в реальный порог Delta-E

---

## 3. ФУНКЦИИ

### 3.1 `replace_colors`

**Главная функция модуля.**

**Вход:**
- `image_rgba`: numpy массив формы (H, W, 4), dtype uint8 — исходное RGBA-изображение
- `color_mappings`: список объектов ColorMapping (from_hex, to_hex)
- `tolerance`: целое число 0–100

**Выход:**
- numpy массив формы (H, W, 4), dtype uint8 — новое RGBA-изображение с заменёнными цветами

**Алгоритм (пошагово):**

1. **Скопировать** входной массив (не мутировать оригинал)
2. **Извлечь** RGB-каналы (H, W, 3) и alpha-канал (H, W) отдельно
3. **Конвертировать** RGB в LAB-пространство через OpenCV `cvtColor` (BGR→LAB, учесть что OpenCV ожидает BGR)
   - Конверсия делается ОДИН раз, результат переиспользуется для всех маппингов
4. **Для каждого маппинга** (from_hex, to_hex):
   a. Конвертировать `from_hex` в LAB-цвет (через промежуточный RGB→BGR→LAB)
   b. Рассчитать `tolerance_threshold` = tolerance * MAX_DELTA_E / 100
   c. Рассчитать Delta-E (CIE76) для КАЖДОГО пикселя: `sqrt((L1-L2)² + (a1-a2)² + (b1-b2)²)`
      - Это vectorized numpy-операция на всей матрице
   d. Построить маску: `mask = (delta_e <= tolerance_threshold)` — булева матрица (H, W)
   e. Если tolerance == 0: `blend_factor` для всех пикселей в маске = 1.0 (точное совпадение)
   f. Если tolerance > 0: `blend_factor = 1.0 - (delta_e[mask] / tolerance_threshold)` — плавный переход
      - Пиксели с delta_e = 0 → blend_factor = 1.0 (полная замена)
      - Пиксели с delta_e = tolerance_threshold → blend_factor = 0.0 (без замены, граница)
   g. Конвертировать `to_hex` в RGB-массив
   h. Для пикселей в маске: `new_rgb = original_rgb * (1 - blend_factor) + to_rgb * blend_factor`
      - Это линейная интерполяция (lerp), устраняет ореолы антиалиасинга
   i. Записать new_rgb в результат
5. **Восстановить** alpha-канал из оригинала (прозрачность НИКОГДА не меняется)
6. **Собрать** RGBA-массив обратно и вернуть

**Обязательная нормализация LAB:** OpenCV LAB-диапазон (L:0-255, a:0-255, b:0-255) кардинально отличается от стандартного CIE LAB (L:0-100, a:-128..+127, b:-128..+127). Перед вычислением Delta-E ОБЯЗАТЕЛЬНА нормализация: L_std = L_cv * 100 / 255, a_std = a_cv - 128, b_std = b_cv - 128. Без нормализации MAX_DELTA_E=50.0 теряет физический смысл, и tolerance будет работать некорректно.

**Управление памятью numpy:** Промежуточные массивы (LAB float32: ~192MB, delta_e: ~64MB, mask: ~16MB, blend: ~64MB = ~336MB на 4000×4000) ОБЯЗАНЫ освобождаться явным del delta_e, mask, blend_factor после каждого маппинга. Использовать np.sqrt(out=...) и другие in-place операции для снижения пикового потребления. Без явного del пиковое потребление удваивается.

**Оптимизация по производительности:** Для каждого маппинга Delta-E вычисляется по всем пикселям (H×W). При 4000×4000 = 16M пикселей и 10 маппингах — 160M операций sqrt. ОБЯЗАТЕЛЕН ранний выход: если маска пуста (ни один пиксель не попал в порог) — пропустить маппинг без вычисления blend. Рекомендуется профилирование на реальных данных 4000×4000.

**Защита от numeric overflow:** При lerp-интерполяции результат может выйти за диапазон 0-255 из-за float rounding. ОБЯЗАТЕЛЬНО np.clip(result, 0, 255).astype(np.uint8) перед записью в выходной массив. Запрещено использовать .astype(np.uint8) без предварительного clip — integer overflow wrapping (256 → 0) создаёт видимые артефакты.

**Критические нюансы:**
- OpenCV работает с BGR, не RGB — конверсия обязательна
- LAB в OpenCV: нормализация ОБЯЗАТЕЛЬНА перед Delta-E (см. выше)
- blend_factor должен быть float32 для точности
- Результат клампить в диапазон 0-255 перед конверсией в uint8

### 3.2 `hex_to_rgb`

**Утилита конверсии.**

**Вход:** строка HEX (например `"#FF0000"`)  
**Выход:** кортеж (r, g, b) — три целых числа 0-255

**Логика:** Парсить строку, убрать `#`, взять по 2 символа на каждый канал, перевести в int base 16.

---

### 3.3 `rgb_to_lab`

**Утилита конверсии одного цвета.**

**Вход:** кортеж (r, g, b)  
**Выход:** кортеж (l, a, b) в формате OpenCV LAB

**Логика:** Создать numpy-массив 1x1x3, конвертировать RGB→BGR→LAB через cv2.cvtColor. Вернуть три float.

## 4. ОПТИМИЗАЦИИ

- ВСЕ операции — numpy vectorized (НЕ попиксельный Python for-loop)
- LAB-конверсия всего изображения — ОДИН раз (кэшируется между маппингами)
- Маска — булева numpy-матрица, индексация через boolean indexing
- blend_factor вычисляется ТОЛЬКО для пикселей в маске (не для всей матрицы)
- Использовать `np.float32` для промежуточных вычислений, финально `np.uint8`

---

## 5. ГРАНИЧНЫЕ СЛУЧАИ

- `tolerance = 0`: только exact match пикселей. blend_factor = 1.0 для совпадений
- `tolerance = 100`: очень широкий захват. Может затронуть соседние цвета
- Пустой список `color_mappings`: вернуть копию оригинала без изменений
- Изображение полностью прозрачное (alpha = 0): цвета всё равно заменяются в RGB-каналах, alpha не трогается
- Перекрывающиеся маски: если два маппинга затрагивают один пиксель — второй маппинг работает поверх результата первого (sequential application). **Инвариант**: LAB-матрица рассчитывается ОДИН раз из оригинала и переиспользуется для всех маппингов. Маска строится по оригинальному LAB, а blend применяется к текущему (уже изменённому) RGB. НЕ обновлять LAB после каждого маппинга — это сломает поведение
- **Порядок маппингов имеет значение:** При перекрывающихся масках результат зависит от порядка маппингов в списке. Это поведение ОБЯЗАНО быть задокументировано в OpenAPI description эндпоинтов preview-replace и create-job. Для будущих версий рекомендуется опция exclusive_mask для взаимоисключающих масок

---

## 6. ТЕСТОВЫЕ СЦЕНАРИИ (для QA)

1. Точная замена одного цвета (tolerance=0): красные пиксели → синие, остальные не тронуты
2. Замена с tolerance=25: антиалиасированные края между красным и белым — плавный переход
3. Множественные маппинги: красный→синий + зелёный→белый в одном вызове
4. Прозрачность: полупрозрачные пиксели (alpha 128) — RGB меняется, alpha остаётся 128
5. Полностью прозрачное изображение: RGB меняется, alpha=0 сохраняется
6. DPI-независимость: функция работает с numpy-массивом, DPI не является её ответственностью
